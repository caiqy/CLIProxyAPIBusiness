# 管理员面板认证文件内容编辑设计

## 1. 背景与目标

当前管理员面板 `web/src/pages/admin/AuthFiles.tsx` 的 **Edit Auth File** 弹窗已经支持编辑以下受管字段：

- `name`
- `key`
- `proxy_url`
- `rate_limit`
- `priority`
- `auth_group_id`
- `is_available`
- 白名单相关字段

后端 `PUT /v0/admin/auth-files/:id` 已支持接收并更新 `content`，但前端界面尚未提供直接编辑认证文件内容的能力。这导致管理员只能通过导入或其他非直观方式修改 `access_token`、`refresh_token`、cookie、过期时间等认证元数据。

本次设计目标是在**不引入新页面、不改变现有编辑路径**的前提下，为认证文件编辑弹窗增加 `content` 原始 JSON 编辑能力。

## 2. 已确认约束

- 编辑方式采用 **原始 JSON 文本框**，不做按 provider 拆分的结构化表单。
- 功能面向 **所有认证文件类型**，不按 provider 特判。
- 继续复用现有 **Edit Auth File** 弹窗，不新增独立页面或独立编辑流程。
- 现有受管字段继续通过独立表单编辑，其中当前明确纳入受管字段的是：
  - `proxy_url`
- 若 JSON 中出现受管字段，前端必须**阻止保存并提示冲突**，而不是静默覆盖或自动删除。
- JSON 需要做**前端实时校验**；JSON 非法或结构不符时必须禁用保存。
- 敏感值（如 `access_token`、`refresh_token`、cookie）按**明文可见可编辑**处理。

## 3. 方案对比

### 方案 A（采纳）：在现有编辑弹窗中直接加入 JSON 文本框

在 `Edit Auth File` 弹窗中新增 `Content (JSON)` 多行编辑区，和现有表单字段一起保存。

**优点**

- 与当前页面流程最一致，学习成本最低。
- 能直接复用现有 `handleEdit` / `handleEditSave` / `PUT /v0/admin/auth-files/:id` 链路。
- 对所有认证类型都通用，不需要 provider 级别适配。
- 管理员可以一次完成基础字段与认证内容的修改。

**代价**

- 弹窗内容会更长，需要保证布局与可读性。
- 需要明确展示“受管字段冲突”的规则，避免用户误解。

### 方案 B：在现有弹窗中增加折叠式高级区

默认隐藏 JSON 区，需要展开后才能编辑。

**不采纳原因**

- 虽然更整洁，但会降低发现性。
- 对当前需求来说，“支持编辑内容”本身是主目标，不适合再藏一层。

### 方案 C：新增独立的内容编辑入口

将认证内容编辑与基础字段编辑拆成两个入口。

**不采纳原因**

- 引入额外路径，增加理解和维护成本。
- 与当前需求“在认证文件编辑界面增加编辑内容功能”的目标不一致。
- 当前不存在必须拆分入口的复杂度压力，属于过度设计。

## 4. 界面与交互设计

### 4.1 弹窗布局

在现有 `Edit Auth File` 弹窗中保留全部已有表单区，并新增一个新的表单块：

- 标题：`Content (JSON)`
- 控件：多行文本框
- 初始值：当前 `file.content` 格式化后的 JSON 字符串（建议 2 空格缩进）

推荐布局为：

1. 名称、Key、Proxy URL、Rate limit、Priority、Auth Group、Status
2. 现有白名单区域
3. `Content (JSON)` 编辑区
4. 错误提示区（按需显示）
5. Save / Cancel 按钮

这样可以保持“基础受管字段在上、原始内容编辑在下”的信息层级，减少“两个地方都能改同一个字段”的误解。

### 4.2 JSON 编辑区行为

- 打开编辑弹窗时，前端将 `file.content` 序列化为格式化 JSON 并填入文本框。
- 文本框允许直接编辑完整认证内容对象。
- 不对敏感字段做脱敏或隐藏，按明文展示。
- 若内容较长，文本框应具备足够高度并支持滚动，以保证可读性。

### 4.3 保存按钮启用条件

保存按钮仅在以下条件全部满足时可用：

1. `name` 非空且合法
2. `key` 非空且合法
3. `Content (JSON)` 是合法 JSON
4. JSON 解析结果是对象（而不是数组、字符串、数字、布尔或 `null`）
5. JSON 不包含受管字段冲突

任一条件不满足时，保存按钮禁用。

## 5. 受管字段边界设计

### 5.1 当前受管字段与冲突键范围

本次需求里，“受管字段”有两层含义，需要明确区分：

1. **弹窗整体受管字段**：包括 `name`、`key`、`proxy_url`、`rate_limit`、`priority`、`auth_group_id`、`is_available` 以及白名单字段。这些字段本来就通过独立表单提交。
2. **JSON 冲突校验键**：仅指那些既可能出现在 `content` 中、又已经被独立表单接管的键。

本次迭代中，**JSON 冲突校验键只有一个：`proxy_url`**。

也就是说：

- 前端保存时始终会提交现有表单字段与 `content`
- 但只有当 `content` 顶层出现 `proxy_url` 时，才触发“受管字段冲突”校验
- `name`、`key`、`rate_limit`、`priority` 等虽然由表单提交，但并不属于本次 `content` 冲突扫描范围

这个规则是本设计的唯一权威定义，测试、验收与实现均应以此为准。

### 5.2 冲突规则

若 JSON 中出现冲突键：

- 前端立即显示错误提示
- 保存按钮禁用
- 不发送更新请求

提示语义应清晰表达：

> `proxy_url` 必须通过独立表单字段编辑，不能在 JSON 中保存。

### 5.3 冲突检测算法

冲突检测规则需要精确定义为：

- **只检查 `content` 顶层键**
- **按精确键名匹配**
- **区分大小写**
- **本次仅匹配 `proxy_url`**
- **不递归扫描嵌套对象或数组内部字段**

示例：

- `{"proxy_url":"http://a"}` → 冲突，禁止保存
- `{"nested":{"proxy_url":"http://a"}}` → 不视为本次冲突
- `{"Proxy_URL":"http://a"}` → 不视为本次冲突

这样可以保证实现与测试预期一致，避免出现“是否需要深度扫描 JSON”的歧义。

### 5.4 不采纳的冲突处理方式

以下方式本次均不采纳：

- **自动忽略 JSON 中的受管字段并以表单为准**：会产生“改了但没生效”的困惑。
- **自动从 JSON 中删除冲突字段再保存**：会造成隐式数据改写，不利于用户理解。
- **允许双写并以某一侧覆盖另一侧**：会模糊边界，增加长期维护成本。

本次采用“**显式阻止 + 明确提示**”策略，保证规则简单且可预期。

## 6. 前后端数据流设计

### 6.1 前端状态

在 `AuthFiles.tsx` 的现有编辑状态基础上，新增与内容编辑相关的前端状态：

- `editContentText`：JSON 文本框当前内容
- `editContentError`：JSON 解析或结构错误提示
- `editContentConflictError`：受管字段冲突提示

也可以将错误收敛为一个统一的派生校验结果，但从职责上至少需要表达：

- 语法错误
- 结构错误
- 冲突错误

### 6.2 打开弹窗

`handleEdit(file)` 在保留现有赋值逻辑的同时，需要新增：

- 从 `file.content` 生成 JSON 编辑初始值
- 初始化 JSON 编辑文本
- 清空内容校验错误状态

初始值规则需要明确为：

1. 若 `file.content` 是普通对象，则使用 `JSON.stringify(file.content, null, 2)` 作为初始值。
2. 若 `file.content` 是数组、字符串、数字、布尔或 `null`，则仍使用其 JSON 序列化结果作为初始值，并立即显示“`content` 必须是 JSON 对象”的结构错误，使管理员可在原值基础上修正。
3. 若出现不可序列化或异常值（理论上极少见），回退为 `{}`，并显示一条非阻塞提示，告知当前内容无法按对象形式回填，已使用空对象初始化编辑区。

这样可以保证弹窗在异常历史数据场景下也有确定行为，而不是打开即崩溃或 silently 丢失上下文。

### 6.3 保存流程

`handleEditSave()` 扩展为以下顺序：

1. 校验现有受管字段（沿用现有逻辑）
2. 对 `editContentText` 执行 `JSON.parse(...)`
3. 校验解析结果必须是对象
4. 校验是否包含受管字段（当前为 `proxy_url`）
5. 若存在任一错误，则中断保存
6. 若全部通过，将解析后的对象作为 `content` 放入更新 payload
7. 继续把现有受管字段与 `content` 一起提交到 `PUT /v0/admin/auth-files/:id`

### 6.4 提交载荷边界

最终请求体由两类字段组成：

- **受管字段**：`name`、`key`、`proxy_url`、`rate_limit`、`priority`、`auth_group_id`、`is_available`、白名单字段
- **原始内容字段**：`content`

边界规则是：

- 受管属性以独立表单值为准
- `content` 不得在**顶层**重复承载冲突校验键 `proxy_url`

这样既复用后端已有接口能力，也能保持前端规则清晰。

### 6.5 保存成功后的本地状态更新

当前页面在保存成功后会局部更新 `authFiles` 列表中的目标项。本次设计要求同步更新：

- `item.content`
- 以及已有受管字段更新结果

这样管理员无需整页刷新，再次打开编辑弹窗时即可看到最新 `content`。

## 7. 错误处理设计

### 7.1 前端即时错误

前端需要覆盖以下错误类型：

1. **JSON 语法错误**
   - 示例：缺逗号、缺引号、括号不闭合
   - 行为：显示错误提示，禁用保存

2. **JSON 结构错误**
   - 示例：输入为 `[]`、`"abc"`、`123`、`true`、`null`
   - 行为：提示 `content` 必须是 JSON 对象，禁用保存

3. **受管字段冲突**
   - 示例：JSON 中出现 `proxy_url`
   - 行为：提示该字段必须通过独立表单编辑，禁用保存

### 7.2 后端兜底

虽然前端已做实时校验，但后端仍是最终边界：

- 非法请求体仍可被后端拒绝
- 后端已有 `content` / `proxy_url` 相关规则继续生效
- 前端不依赖“后端报错后再修正”的交互，但后端仍保留数据完整性保护职责

### 7.3 失败时界面行为

若更新请求失败：

- 不关闭编辑弹窗
- 不更新本地列表项
- 保留当前输入内容，便于管理员继续修正
- 显示一条**用户可见**的提交失败提示（可采用弹窗内错误横幅或现有错误展示方式，但必须对管理员可见）
- 同时保留控制台错误记录，便于排查

## 8. 测试设计

优先沿用当前 `AuthFiles.tsx` 相关测试风格，在前端补充用例。

### 8.1 前端测试范围

1. **打开弹窗时回填内容**
   - `file.content` 被格式化后显示在 JSON 文本框中

2. **合法 JSON 可提交**
   - 修改 JSON 后点击保存
   - 请求体包含 `content`
   - 仍同时保留现有受管字段

3. **非法 JSON 禁止保存**
   - 显示语法错误提示
   - 保存按钮 disabled

4. **非对象 JSON 禁止保存**
   - 输入数组、字符串、数字、布尔或 `null`
   - 显示结构错误提示
   - 保存按钮 disabled

5. **受管字段冲突禁止保存**
   - JSON 顶层包含 `proxy_url`
   - 显示冲突提示
   - 不发送请求

6. **异常初始内容可稳定打开**
   - `file.content` 为非对象时，编辑区仍可打开
   - 初始值可见，且立即展示结构错误提示

7. **保存成功后本地状态更新**
   - 列表中的目标项 `content` 被同步更新
   - 再次打开弹窗时展示更新后的 JSON

### 8.2 后端测试

本次功能不以新增后端接口为目标，因此不要求围绕该需求新增专门后端设计；但若实现过程中触及后端校验行为，应保证现有 `auth_files` 更新路径相关测试不被破坏。

## 9. 验收标准

1. 管理员可以在现有 **Edit Auth File** 弹窗中直接编辑认证文件 `content`。
2. JSON 编辑区对所有认证文件类型可用。
3. JSON 非法时，前端实时提示错误并禁用保存。
4. JSON 虽合法但不是对象时，前端阻止保存。
5. JSON 顶层出现 `proxy_url` 时，前端阻止保存并提示冲突。
6. `proxy_url` 只能通过独立表单字段修改。
7. `file.content` 为非对象时，编辑弹窗仍能稳定打开并给出明确结构提示。
8. 更新请求失败时，管理员能看到明确失败提示，且当前输入不会丢失。
9. 保存成功后，列表状态与再次打开弹窗时看到的内容保持一致。
10. 继续复用现有 `PUT /v0/admin/auth-files/:id` 更新链路，不新增新页面或新接口。

## 10. 非目标

- 不为不同 provider 设计结构化内容表单
- 不新增独立的认证内容编辑页面
- 不修改认证文件导入流程
- 不改变现有白名单能力边界
- 不对敏感字段做默认脱敏、折叠或二次确认
- 不在本次设计中引入代码编辑器级能力（如语法高亮、格式化按钮、diff 预览）

## 11. 实施边界说明

本次设计聚焦于**管理员面板认证文件编辑弹窗中的内容编辑能力**。如未来再增加新的受管字段，应遵循相同原则：

- 受管字段通过独立表单编辑
- JSON 中出现受管字段时显式报错
- 避免形成双写同义字段

这样可以让认证文件编辑能力在持续演进时保持一致的数据边界与交互预期。
