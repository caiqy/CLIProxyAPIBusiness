# 管理员面板模型映射批量删除设计

## 背景

当前管理员面板的“模型映射”页面已经具备：

- 行选择（`selectedIds`）
- 批量启用
- 批量停用
- 单条删除

但尚不支持“批量删除所选”。这导致管理员在清理大量模型映射时需要逐条删除，效率较低。

## 目标

在管理员面板“模型映射”页面增加“批量删除所选”能力。

具体要求：

1. 仅在前端页面实现，不新增后端接口。
2. 复用现有单条删除接口 `DELETE /v0/admin/model-mappings/:id`。
3. 删除策略采用“逐条删，失败保留”。
4. 白名单、模型映射后端语义、权限模型不做扩展性变更。

## 非目标

1. 不新增后端批量删除 API。
2. 不修改单条删除接口语义。
3. 不修改模型映射数据结构。
4. 不调整页面筛选、搜索、排序行为。

## 方案对比

### 方案 A（采用）

在 `web/src/pages/admin/Models.tsx` 中新增批量删除按钮与处理逻辑：

- 使用现有 `selectedIds`
- 确认后通过 `Promise.allSettled` 逐条调用单条删除接口
- 成功删除的条目从本地列表移除
- 删除失败的条目保留在列表和选中状态中

优点：

- 改动小
- 与现有批量启用/停用风格一致
- 不新增后端成本

缺点：

- 多选较多时会产生多次请求

### 方案 B

逐条删除且每次删除后都重新 `fetchData()`。

缺点：

- 请求更多
- UI 易闪烁
- 不必要地放大后端压力

### 方案 C

新增后端批量删除接口。

缺点：

- 超出当前需求
- 需要同步补权限、后端测试、错误明细协议

## 最终设计

采用方案 A。

### 前端交互

在现有批量启用/停用按钮旁新增一个红色“删除”按钮：

- 按钮仅在用户具备 `DELETE /v0/admin/model-mappings/:id` 权限时可用
- `selectedCount === 0` 时按钮禁用
- 标题文案复用现有语义：`Bulk delete selected`

### 确认弹窗

点击后弹出确认框，复用现有批量操作弹窗模式：

- 标题：`Delete Model Mappings`
- 内容：`Are you sure you want to delete {{count}} model mapping(s)?` 或继续复用通用模板 `Are you sure you want to {{action}} {{count}} model mapping(s)?`
- 确认按钮：`Delete`
- `danger: true`

### 删除执行语义

确认后：

1. 取当前 `selectedIds`
2. 对每个 id 调用 `DELETE /v0/admin/model-mappings/:id`
3. 使用 `Promise.allSettled` 收集结果

处理结果：

- 成功删除：从 `mappings` 中移除
- 删除失败：保留在 `mappings` 中
- 选中状态更新为“仅保留失败项”
- 若全部成功，则清空选中状态
- 最终关闭确认框

### 错误处理

- 失败请求仅记录到控制台（保持和现有页面风格一致）
- 不中断其他条目的删除
- 不因单条失败回滚已成功删除的条目

## 状态与数据流

现有页面已经维护：

- `mappings`
- `selectedIds`
- `confirmDialog`

因此不需要新增复杂状态；仅增加一个 `handleBulkDelete()`：

- 输入：`selectedIds`
- 输出：更新后的 `mappings` 与 `selectedIds`

## 权限边界

严格沿用现有权限：

- 仅使用 `canDeleteMapping`
- 不新增权限点

## 测试策略

优先补前端页面测试（Vitest + React Testing Library）：

1. 选中多条后，“删除”按钮变为可用
2. 点击后显示确认弹窗
3. 全部删除成功时：
   - 列表移除对应条目
   - 选中状态清空
4. 部分删除失败时：
   - 成功条目移除
   - 失败条目保留
   - 选中状态仅保留失败条目

若当前页面测试基建不足，至少保证：

- `npm test`
- `npm run build`

## 风险与控制

### 风险

1. 本地状态更新不正确，导致“已删条目仍显示”或“失败条目被误清空”
2. 过滤视图下批量删除后，选中状态与列表不同步

### 控制

1. 仅基于返回结果更新 `mappings`
2. 删除后按成功/失败 ID 集合更新 `selectedIds`
3. 复用现有 `selectedIds` 与筛选逻辑，不引入额外状态分叉

## 结论

本次功能应以最小改动方式落地：

- 前端单页实现
- 复用现有删除接口
- 采用“逐条删，失败保留”
- 不新增后端接口与权限

这样可以在保持现有架构稳定的前提下，显著提升管理员批量清理模型映射的效率。
